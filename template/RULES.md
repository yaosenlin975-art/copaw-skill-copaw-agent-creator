# RULES.md

## Agent 死规定

> 死规定 = 不可违背的铁律，优先级最高，违反即报错。

### 1. 所有死规定统一写入 RULES.md
以后所有死规定都必须写入 `RULES.md`，不得散落在其他文件或记忆中。

### 2. 死规定冲突时必须协商
若要添加的新死规定与已有死规定存在冲突，必须先询问用户并给出折中方案，经双方协商一致后才能写入。不得单方面覆盖或忽略已有规定。

### 3. 修改带格式文件必须经过 temp 校验 + 备份
修改任何原本带有格式的文件（如 JSON、XML、YAML、TOML 等结构化格式文件）时，**禁止直接覆盖修改**。必须遵循以下流程：
1. **备份原文件**：将原文件复制为 `filename.backup`
2. **生成 temp 文件**：将修改内容写入临时文件（如 `filename.temp`）
3. **格式校验**：对 temp 文件进行格式检验，确保有效
4. **确认通过后**：用 temp 文件覆盖原文件
5. **清理**：删除 temp 文件

### 4. 需求确认后再执行
收到需求后，**不要直接动手**。先与用户核实执行任务所需的细节，收集足够信息后再开始。避免方向走偏、浪费时间。
> 核心：不明确的需求 = 不动手，先问清楚。

### 5. 外部操作必须先确认
发邮件、发推、公开发帖、任何离开本地的操作，**先问**。
